## 1. 先告诉你这篇综述总体在讲什么

### 1.1 这篇综述关注的核心主题是什么

这篇综述关注的核心主题是 **Personalized LLM-powered Agents（个性化大模型智能体，PLAs）**。它讨论的不是“怎样让大模型说话更像某个用户”这么窄的问题，而是：**当 LLM 变成一个会推理、会规划、会调用工具、会和环境互动的 agent 之后，如何让它在长期交互中持续对齐某个具体用户，而不是只提供泛化的、平均意义上的帮助。** 综述开篇就强调，随着 agent 开始承担长期、多轮、跨任务的协作，个性化会从表层生成问题变成系统级问题。 ([arXiv][1])

### 1.2 这个领域主要在研究什么问题

这个领域主要研究：**agent 如何理解“这个用户是谁”、如何记住与这个用户有关的信息、如何把这些用户信息真正用于决策、以及如何把决策落实成符合用户偏好的行动与结果。** 综述把这些问题系统化成四个核心能力：**profile modeling（画像建模）**、**memory（记忆）**、**planning（规划）**、**action execution（动作执行）**。作者的一个关键观点是，个性化不是某个单独模块的能力，而是这四个能力协调运作的结果。 ([arXiv][1])

### 1.3 为什么会有这篇综述，这个方向为什么值得研究

之所以需要这样一篇综述，是因为此前关于 agent 的综述很多，但往往只看某个局部：比如只讲用户建模、只讲记忆、只讲 planning，或者只讲一般 LLM agent，而没有把“个性化如何沿着整条 agent pipeline 传播和落地”作为统一主题来讲。作者明确指出，过去的研究视角是碎片化的，而现实中一个真正可用的个性化 agent，必须把用户信号从输入、内部状态、规划决策一直传到外部执行结果。这个方向值得研究，是因为教育、医疗、推荐、研究辅助、委托代理这些真实应用里，**“同一个问题对不同用户并没有同一个正确答案”**，系统必须具有用户条件化（user-conditioned）的能力。 ([arXiv][1])

### 1.4 综述整体按什么逻辑展开

这篇综述整体不是按年份排论文，也不是按应用领域排，而是按一种 **capability-oriented taxonomy（能力导向分类）** 来展开。前面先讲 foundations：什么是 agent、什么是 personalized agent、用户数据有哪些类型、偏好有哪些类型；然后用图 4 和表 1 给出四大能力框架；接着正文依次展开四大模块：profile modeling、memory、planning、action execution；之后讲 evaluation，再讲 applications，最后总结 open problems and future directions。换句话说，文章的组织逻辑是：**先定义个性化 agent 的输入与偏好，再按“表示—保留—推理—落实”的链路来讲研究工作。** ([arXiv][1])

## 2. 梳理这篇综述涉及的核心问题

### 2.1 核心问题一：如何表示用户

这是最基础的问题。因为 agent 想个性化，首先要知道“用户是谁”。但“用户是谁”并不是一个固定字段，它可能包括人口统计信息、历史行为、长期价值观、短期偏好、表达风格、任务偏好等等。综述在 foundations 里把用户数据分成 **historical data（历史数据）** 和 **interaction data（交互数据）**：前者决定“这个用户大体上是谁”，后者决定“当前这次互动里用户现在想要什么”。同时它又把偏好分成 **explicit vs. implicit**（显式/隐式）以及 **behavioral vs. topical**（行为型/主题型）两条轴。也就是说，用户建模既要处理长期稳定特征，也要处理任务当下的临时意图。 ([arXiv][1])

### 2.2 核心问题二：如何让个性化跨轮次、跨任务持续存在

光知道用户当下偏好还不够，agent 还要能“记住”。所以第二个核心问题是：**如何把用户相关的信息存起来，并在之后正确取出来。** 这就进入 memory 问题。综述指出，profile modeling 更多刻画的是相对稳定的、抽象的用户理解，而 memory 负责保存跨轮次、跨会话的细粒度信息，承担“长期连续性”的角色。这里的难点是：用户偏好不同维度变化速度不同，系统既要快速更新短期偏好，又要保留长期稳定特征，不能把所有旧信息都丢掉，也不能把过时信息一直留下。 ([arXiv][1])

### 2.3 核心问题三：如何把用户信息转化成决策

第三个问题是 planning。用户信息不只是拿来“修饰回答语气”，而是要真正影响 agent 的决策过程。举例说，同样是旅行规划，不同用户对预算、舒适度、风险、节奏、景点偏好可能完全不同；同样是研究助手，不同用户对文献广度、深度、解释方式、写作风格的要求也不同。综述因此把 planning 看成个性化落地的关键中间层：它负责把用户画像、检索到的记忆、当前任务约束和交互反馈，转化成**用户条件化的子目标、约束与权衡策略**。 ([arXiv][1])

### 2.4 核心问题四：如何把个性化真正体现在动作与结果上

第四个问题是 action execution。即使规划阶段“想对了”，如果最后工具没选对、参数没填对、执行失败没修、结果排序或呈现方式不符合用户偏好，个性化也并没有真正实现。综述特别强调，个性化执行与一般任务执行的区别在于：**它不仅要完成任务，还要在可行的多个执行方案中，选出最符合该用户偏好的那个。** 因此，执行层要处理 preference-aware tool selection、参数补全、约束放松、故障修复、结果重排和主动补充建议。 ([arXiv][1])

### 2.5 这些问题之间是什么关系

我基于综述做一个归纳：这四个问题构成一个闭环。**profile modeling** 负责构造“用户是谁”的抽象表示，**memory** 负责把这种表示与交互事实持续保存并更新，**planning** 负责把这些用户信号转化成当前任务的决策逻辑，**action execution** 负责把决策变成对外部世界的动作与结果。然后，执行结果和用户反馈又反过来更新 profile 和 memory，形成 Figure 1 所展示的闭环。也就是说，这不是四个孤立模块，而是一条从用户信号到个性化结果、再回流到用户建模的循环链。 ([arXiv][1])

### 2.6 哪些是基础问题，哪些是延伸问题

基础问题是：**如何建模用户、如何记忆用户、如何基于用户做规划、如何在执行中贯彻用户偏好。** 延伸问题是在这四个基础能力之上出现的：比如如何在低交互用户上泛化，如何在高风险场景下做安全个性化，如何让系统在隐私约束下依然保持个性化质量，如何在多领域、多工具、多时段上保持稳定一致。作者在 open problems 里把这些提炼为 decision-critical user modeling、continual personalization、generalization、evaluation、privacy、efficiency 等更高层问题。 ([arXiv][1])

### 2.7 目前这个领域的主要难点

综述点出的主要难点包括：用户信号稀疏、噪声大、矛盾多；用户偏好有的长期稳定、有的短期波动；系统很难判断哪些用户属性对当前决策真正关键；长期记忆会堆积冗余、陈旧或幻觉信息；规划中要在“少打扰用户”和“别误解用户”之间平衡；执行层要把抽象偏好转成具体参数；评测又很难，因为个性化成功与否不是单一客观答案，而是“是否对这个用户更有用”；此外还有隐私泄露、过度保守、成本过高等部署问题。 ([arXiv][1])

## 3. 系统介绍综述里提到的研究工作

这一部分是重点。我按综述的四大能力来分组，并在每一类内部讲它们为什么会从 A 发展到 B，再到 C。

---

### 3.1 第一大类：Profile Modeling——先弄清楚“这个用户是谁”

#### 3.1.1 这一类工作的基本思路

这一类工作的基本思路是：**先把原始用户信号整理成一个可供 agent 使用的内部表示。** 如果没有这个表示，后面的记忆、规划、执行都只能围绕“平均用户”做。综述把这一类工作分成两部分：**user profile modeling** 和 **agent role definition**。前者回答“用户怎么表示”，后者回答“agent 应该以什么角色来面对这个用户”。 ([arXiv][1])

#### 3.1.2 它想解决什么问题

它想解决的是个性化的入口问题：**用户信息太杂乱，怎么提炼成对决策有用的表示？** 如果系统只拿到零散的聊天记录、点击行为、反馈语句，很难直接驱动 agent 行为，所以需要一个中间层把它们变成 persona、reward weights、文本摘要、偏好向量等更稳定的形式。 ([arXiv][1])

#### 3.1.3 为什么会从一种方法发展到另一种方法

最早、最直观的做法是 **persona-based modeling**：给用户做一个整体画像，比如“偏正式表达、重视安全、喜欢结构化回答、偏向省钱旅行”。这种方法好理解、好解释，但缺点是太“整体化”，在具体情境下不够细。于是后来出现 **response-based modeling**：不再试图先定义一个完整人格，而是直接去建模“这个用户会更喜欢 A 回答还是 B 回答”。这是一种更细粒度、更贴近实际满意度的建模方式。进一步地，研究者发现，光有用户表示还不够，agent 自己也得根据用户调整扮演方式，于是又发展出 **agent role definition**，让 agent 的 persona、语气、主动性、自治程度也随着用户而变。 ([arXiv][1])

#### 3.1.4 它通常怎么做，核心机制是什么

* **Persona-based**：从自我描述、历史行为、历史交互中提取相对稳定的长期特征，形成结构化属性或自然语言 persona summary，再作为 prompt 或约束输入下游模型。
* **Response-based**：从用户对候选输出的偏好中学习一个更细粒度的偏好表示，比如把个体偏好因式分解成共享特征基 + 用户特定权重，或者用自然语言总结偏好维度。
* **Role definition**：让 agent 不只是“知道用户”，而是“知道自己该怎样和这个用户互动”，这可以是替用户执行（user-substituting），也可以是在合作中动态调整自己的角色（user-adaptive）。 ([arXiv][1])

#### 3.1.5 优势、局限、与其他类别的区别

Profile modeling 的优势是：它提供了全系统的起点，后续所有模块都可受益。它的局限是：如果建模过静态，会跟不上用户变化；如果建模过细，又容易过拟合或引入噪声。它和 memory 的区别在于，**profile 更像“抽象理解”，memory 更像“事实仓库”**；和 planning 的区别在于，**profile 解决“知道用户”，planning 解决“基于用户做决定”**。 ([arXiv][1])

#### 3.1.6 代表性工作及其演进脉络

* **AlignXpert**：用高维偏好空间来表示用户，说明 persona 可以不只是几句文字，而能映射到更系统的心理/对齐维度。
* **FSPO**：通过 user-description chain-of-thought 生成更细致 persona，说明研究开始从粗略画像走向细粒度画像。
* **RLPA、Wang and Yang、RGMem**：从一次性画像，走向跨轮交互中持续更新画像。
* **RLPF**：强调 persona summary 要可读、可控，并且与下游效果闭环优化。
* **CURIO**：再进一步，不是被动等用户告诉你，而是主动推断 latent user types。
* **RFM、PReF、LoRe**：把用户偏好建模成 response-level 的权重或 reward 参数，适合稀疏反馈场景。
* **Janus、PLUS**：用自然语言结构化地表达偏好，让 response-based 模型也有可解释性。
* **LD-Agent、PersonaAgent、PersonaX、RPP**：把研究推进到“agent 角色也要个性化适配用户”。

我基于综述的概括是：这条线的发展逻辑是 **“从静态画像 → 动态画像 → 从整体用户描述到具体响应偏好 → 再到 agent 自身角色的个性化重配置”**。 ([arXiv][1])

### 3.2 第二大类：Memory——让 agent 不只是知道用户，还能长期记住用户

#### 3.2.1 这一类工作的基本思路

Memory 的基本思路是：**把与用户有关的信息长期保存，并在合适的时候取出来参与当前决策。** 综述明确区分了 internal memory 和 external memory，但重点讨论的是长期外部个人记忆，因为这是个性化 agent 中研究最多、也最灵活的一类。 ([arXiv][1])

#### 3.2.2 它想解决什么问题

它要解决的是 profile modeling 解决不了的问题：profile 更适合高层稳定特征，而现实交互中有大量细节、事实、事件、偏好变化、短期目标，需要被长期保存。没有 memory，agent 只能“每次重新认识用户”；有了 memory，agent 才能表现出连续性。 ([arXiv][1])

#### 3.2.3 为什么会出现文本记忆与结构化记忆之分

早期做法很直接：直接存整段对话或执行轨迹。但这种方法冗余大、难检索，于是研究开始把原始历史压缩成更适合下游使用的表示。这样就分出了两条路线：
一条是 **textual memory**，用自然语言摘要和片段来存；
另一条是 **structured memory**，把信息组织成树、图、triples、层次结构。
前者更贴近 LLM 输入输出格式，后者更利于组织复杂关系和长距离依赖。 ([arXiv][1])

#### 3.2.4 文本记忆：怎么发展起来的

文本记忆最初的问题是：你到底按什么粒度存？如果按 turn 存，太碎；按整场会话存，又太粗。于是研究先从 **分段（segmentation）** 开始优化。

* **SeCom、RMM**：开始做语义一致的分段，而不是机械按轮次切。
* **Nemori**：更进一步，显式检测 episode boundary，尽量保持用户意图的完整性。
  之后大家发现，光分段还不够，还得让每段记忆更丰富，于是加入标签、时间戳、topic、personality 等辅助属性。
* **CAIM**：给记忆条目加 tags、inductive thoughts、timestamps。
* **MemInsight**：从对话中提取更结构化的语义属性，帮助后续使用。
  再往后，一部分研究开始模仿人类记忆系统，区分 semantic 和 episodic。
* **MIRIX**：用层级化多组件记忆架构。
* **Mem-**：把 core / semantic / episodic memory 联合起来，并用 RL 管理。

这条线说明文本记忆经历了 **“原始日志 → 语义分段 → 带属性的摘要 → 多层级认知化记忆”** 的演进。 ([arXiv][1])

#### 3.2.5 结构化记忆：为什么出现，怎么做

结构化记忆出现的原因是：很多个性化信息不只是“几段文字”，而是有关系、有层次、有抽象链路。比如用户喜欢某类电影，这又和导演、演员、题材、情绪类型等关系有关。如果只靠文本检索，很容易只抓到表层相似度，抓不住跨条目关系。于是有了树和图。

* **MemTree**：把新信息按语义相似度插入树中，并同步更新祖先摘要，代表层级化树结构。
* 一些 graph-based memory 会用 triples、nodes、edges 来表示信息关系，并在冲突出现时做关系失活或更新。
* **SGMem、LiCoMemory**：是更轻量的结构化方案，说明研究后来开始关注“结构很强，但不能太重”，要兼顾效率。

这条线本质上是把 memory 从“内容仓库”推进成“关系化知识结构”。 ([arXiv][1])

#### 3.2.6 Memory update：为什么从简单覆盖发展成显式操作和推理更新

记忆最难的不是存，而是更新。因为用户偏好会变，还会互相矛盾。综述把 textual memory 的更新分成 **similarity-driven** 和 **inference-guided** 两类；structured memory 则分成 **similarity-driven** 和 **reasoning-/agentic-driven**。

* 早期如 **RMM、CAIM**，更多是在语义相似基础上合并、改写、替换。
* **Mem0** 很关键，因为它把更新显式化成 **ADD / UPDATE / DELETE / NOOP**，相当于把“记忆维护”从隐式摘要，推进成可操作的状态转移。
* **Memory-R1** 再往前一步，不只定义操作，还学习一个更新策略。
* **PREMem** 则在存入前先推理主题演化趋势，再决定如何插入。

这个发展说明研究从“把新东西堆进去”走向“像数据库管理员一样管理记忆”。 ([arXiv][1])

#### 3.2.7 Personalized retrieval：为什么从相似度检索发展到结构检索和策略检索

存好之后，还要取。综述把 personalized retrieval 分成三类：

* **content-based retrieval**：最传统，按语义或词汇相似度找。
* **structure-aware retrieval**：利用图、树、层级关系找。
* **policy-guided retrieval**：不再把检索看成固定匹配，而是把它看成一个用户/任务条件化的决策过程。

这条演化特别重要。
第一代方法如 **RAPTOR、MemTree** 仍以相似度为主，只是支持多粒度层级。
第二代如 **AriGraph、Zep、MemGAS、AssoMem** 开始显式利用图结构、节点距离、关联路径。
第三代如 **PBR、PersonaRAG、UniMS-RAG、Persona-DB、OCG-Agent** 则更激进：它们会先改写查询、决定用哪类检索工具、甚至做结构化字段补全。也就是说，检索从“被动搜”变成“主动找”。 ([arXiv][1])

#### 3.2.8 这一类工作的优势、局限与和其他路线的区别

Memory 路线的优势是能提供长期连续性，是个性化 agent 真正“像认识你”的关键。局限是：长期系统容易引入陈旧、冗余、幻觉和错误压缩；更新策略难；检索不当反而会把噪声带入当前推理。它与 profile 的区别是，profile 是更高层、抽象的用户建模；它与 planning 的区别是，memory 负责提供材料，planning 负责用这些材料做决策。 ([arXiv][1])

### 3.3 第三大类：Planning——把用户信号变成“个性化决策过程”

#### 3.3.1 这一类工作的基本思路

Planning 的基本思路是：**把用户相关信息变成当前任务中的目标、约束和决策路径。** 综述把它分成两大类：**One-shot Planning** 和 **Feedback-driven Planning**。前者是一次性把计划生成出来，后者是把计划当成暂定方案，再边问边改。 ([arXiv][1])

#### 3.3.2 它想解决什么问题

它想解决的是“知道用户 ≠ 会为用户做决定”这个问题。你可以知道一个用户喜欢轻松旅行、讨厌赶行程、预算有限，但如何把这些偏好转成一条可执行的旅行 itinerary，这就是 planning 的职责。综述指出，planning 的两个核心角色是：一，把用户建模真正转成下游决策；二，在交互过程中不断逼近真实意图。 ([arXiv][1])

#### 3.3.3 One-shot planning 为什么会出现，怎么做

One-shot planning 的出现，是因为很多场景不适合高频打扰用户。用户给出一个目标后，agent 希望尽量一轮推理就给出一个完整计划。综述把它又分成两层：

* **user signal grounding**：先把用户信号变成 planning 可用的先验。
* **internal refinement**：先生成一个计划，再在同一推理过程中自我批判、自我修正。

在 grounding 里：

* **PRIME** 很有代表性，它结合 dual-memory 和 Personalized Thought Process，在生成 plan 之前先合成一个“个性化思维轨迹”，让历史经验和稳定信念显式影响当前决策。
* **PersonaAgent** 则从统一记忆库中检索 persona，动态构造 system prompt，维持跨会话计划一致性。
* recommendation、dialogue 里的很多工作也证明：结构化用户记忆检索能显著改善 one-shot 决策质量。

在 internal refinement 里：

* 一般性机制可以借用 **Reflexion** 或 critique-and-revise 范式。
* **TravelPlanner++、PTS**：在旅行规划中迭代修正中间 itinerary，以适配用户隐式长程偏好。
* **PREFINE**：构造 pseudo-user critic 和 user-specific rubrics，对候选计划做用户视角的打分与修订。
* **IoTGPT**：在具身命令执行中，用模拟环境先测试中间命令序列，再修。

所以 one-shot planning 的主线是：**先把用户信号压缩进“决策前提”，再在单轮推理里做内部自修正。** ([arXiv][1])

#### 3.3.4 Feedback-driven planning 为什么会出现，怎么做

One-shot 方法的缺点是：很多任务里用户意图天生不完整、会变化、会含糊，单轮生成容易拍脑袋。于是出现了 feedback-driven planning，把计划当作“一个待验证的假设”。综述把它分成三种反馈来源：

* **clarification seeking**：主动问用户补信息。
* **user revision**：用户直接修改中间计划。
* **non-user feedback**：由其他 agent、环境模拟结果等非用户信号推动重规划。

代表性工作包括：

* **Ask-before-Plan**：把澄清与计划解耦，先解决关键不确定性再做计划。
* **AIPOM**：把计划表示成可编辑结构，让用户能直接改中间方案，系统再同步更新约束和优先级。
* **Dango、PlanFitting**：在数据处理流程、健身计划等场景中，把用户修订纳入循环。
* **VAIAGE**：用多智能体协商路线可行性和推荐结果，本质上是借“别的 agent”给 planning 加反馈。
* **Collab-Rec**：处理多方利益相关者的协同推荐规划。
* **SceneWeaver**：先执行/模拟中间结果，再根据中间结果回头修 plan。

这条路线解决的是：**当用户偏好不完整或计划可行性不确定时，不要一次性拍板，而是逐步收集信息、修正方案。** ([arXiv][1])

#### 3.3.5 优势、局限与和其他类别方法的区别

One-shot 的优势是快、连贯、低打扰；缺点是在信号稀疏或偏好漂移时脆弱。Feedback-driven 的优势是更稳、更贴近真实需求；缺点是交互成本高、延迟高。它和 memory 的区别在于，memory 主要解决“保存和提供用户信息”，planning 解决“用这些信息做权衡和决策”；它和 action execution 的区别在于，planning 是决策设计层，execution 是实施层。综述明确指出，未来重要方向是把二者融合成 **uncertainty-aware planning**：有把握时快做，没把握时再多问、多算、多改。 ([arXiv][1])

### 3.4 第四大类：Action Execution——让个性化不只停留在脑子里，而真的体现在行动上

#### 3.4.1 这一类工作的基本思路

Action execution 的基本思路是：**把个性化决策落实成工具调用、环境操作和最终结果交付。** 综述把它分成 pre-action 和 post-action 两阶段。前者关注“做什么以及怎么参数化做”，后者关注“做完后怎么修、怎么呈现结果”。 ([arXiv][1])

#### 3.4.2 它想解决什么问题

它要解决的是：很多任务里，多个动作都能完成“客观任务”，但只有一部分动作符合当前用户的偏好。比如同样能订到酒店，但有人重视便宜，有人重视位置，有人重视退改灵活；同样能完成网页任务，但有人偏向少步骤、有人偏向高确定性。个性化 execution 就是要让 agent 在“都能做成”的多个方案中，选最适合这个用户的那个。 ([arXiv][1])

#### 3.4.3 Pre-action：从 action policy 到 action grounding

在 **action policy** 层，研究核心是“选哪种执行方式”。

* **PEToolLLaMA**：把个性化工具学习形式化，从交互历史中学 preference-aware tool selection。
* **PUMA**：在 web-agent 环境里做偏好感知的动作选择。
* **ToolSpectrum**：同时考虑 user profile 和 environment context。
* **ValuePilot**：更抽象，研究按个体价值观来做 action-level 决策。
* **ColorAgent**：当信息不完整时，先向用户澄清，再决定执行。

在 **action grounding** 层，问题变成“这个动作具体怎么落参”。

* **AWARE-US**：非常典型，它把 grounding-time infeasibility 看成“偏好感知的查询修复问题”。如果用户约束冲突，不应随便按默认规则放松，而应优先放松用户最不在意的那个约束。
* Huang 等工作的 personalized tool invocation 研究：关注当必要参数缺失时，系统能否根据用户画像自动补足而不误解用户。

我基于综述的总结是：pre-action 这条线的发展逻辑是 **“先解决工具/动作选什么，再解决具体参数怎么填，最后开始处理含糊与不可行的约束修复。”** ([arXiv][1])

#### 3.4.4 Post-action：为什么要从“做完就结束”发展到纠错和结果实现

很多 agent 工作以前到“工具返回结果”就算结束，但对个性化 agent 来说，这远远不够。因为执行结果可能与用户真正想要的还有偏差。于是 post-action 出现两类问题：

* **execution-time correction**：执行失败或效果不好，能否不推翻整个计划就局部修复。
* **outcome realization**：就算多个结果都客观可行，最后应该用什么方式交给用户。

综述指出，这一块明确做“个性化恢复”的工作还不多，但相关研究已经提供了积木：

* tool overuse under diminishing returns 的研究，以及
* structured reflection over tool-interaction failures
  都可以成为未来个性化 recovery 的基础。

在 outcome realization 上，代表性工作更清晰：

* **PEAR、MIR**：做 personalized re-ranking，让最后结果排序更符合用户效用。
* **WARPP**：在运行时根据用户属性调整 workflow branch，减少无关或错误结果。
* **ETAPP**：把 proactivity 也当成 personalization signal，衡量 agent 是否会主动给出额外建议或后续动作。

这说明 execution 研究已经从“正确完成任务”进一步发展到“如何以用户更满意的方式完成任务并交付结果”。 ([arXiv][1])

#### 3.4.5 这一类工作的优势、局限与和其他类别的区别

执行层的最大价值是，它让个性化真正可观察。前面三层再聪明，如果最后动作和结果没体现出来，用户就感受不到个性化。它的局限也最现实：工具、环境、领域差异很大，偏好在执行时又常常不以显式文本出现，而是要被翻译成参数和约束，所以泛化很难。和 planning 的区别是，planning 决定“应当怎么做”，execution 决定“实际怎么做成”。综述也明确说，这一块目前研究相对少，是未来的重要增长点。 ([arXiv][1])

## 4. 详细解释综述中的代表性方法 / 模型 / 框架

下面我挑一些在这篇综述里位置最关键、最能代表不同路线的工作来展开讲。

### 4.1 AlignXpert / FSPO / RLPF / CURIO：画像建模这条线的代表

这些工作共同代表 **persona-based profile modeling**。它们的背景是：早期个性化系统往往只有粗粒度 persona，无法支撑复杂决策。

* **AlignXpert** 的贡献是把用户偏好放进高维、心理和 alignment 相关的空间里，说明 persona 不必只是标签，而可以是系统化特征空间。
* **FSPO** 往前一步，通过 user-description chain-of-thought 生成更细的 persona，让画像更细粒度。
* **RLPF** 则发现，画像不能只追求“丰富”，还必须可读、可控，并且要与下游任务效果挂钩，于是它用反馈回路优化 human-readable user summaries。
* **CURIO** 代表更主动的一步：不是等着收集足够数据，而是在多轮交互中主动推断 latent user type。

它们在研究脉络中的位置，是把 user profiling 从“静态标签化描述”推进到“细粒度、可更新、主动推断的个体建模”。 ([arXiv][1])

### 4.2 RFM / PReF / LoRe / Janus / PLUS：从画像走向“偏好函数”

这组工作代表 **response-based modeling**。它们出现的背景是：persona 虽然好解释，但对实际输出满意度不够精确。

* **RFM、PReF、LoRe** 的核心想法是，把个体偏好表示成“共享 reward 特征基 + 用户特定权重”，这样即使用户反馈不多，也能快速估计其偏好方向。
* **Janus** 把偏好维度做成层级结构，从粗类到细粒度价值描述，提升可解释性。
* **PLUS** 则用文本化偏好总结来条件化 downstream reward modeling 和生成。

这类方法的改进点在于：它们不再只问“用户大概是什么样的人”，而是更直接地问“用户到底更喜欢哪类输出”。它们适合反馈丰富、需要精细对齐的场景，但也依赖足够可靠的响应级反馈。 ([arXiv][1])

### 4.3 PersonaAgent / PersonaX / LD-Agent / RPP：让 agent 自己也变得“有角色感”

这组工作对应 **agent role definition**。它们的背景是：仅仅建模用户还不够，agent 自己如何表现、如何说话、如何设定自治程度，也应该和用户匹配。

* **LD-Agent** 做的是用户和 agent 的双向 persona 建模，并维护长期 persona bank。
* **PersonaAgent** 用 inferred persona 动态构造 system prompt，强调测试时动态适配而非重训练。
* **PersonaX** 则结合离线多 persona profiling 与在线 profile retrieval。
* **RPP** 用历史用户信息和任务指导生成 personalized discrete prompts。

这条线在大脉络中的意义是：个性化不再只是“给同一个 agent 喂不同用户信息”，而是“agent 这个合作对象本身也会因人而异”。 ([arXiv][1])

### 4.4 Mem0 / Memory-R1 / MemTree / MIRIX：长期记忆从“存摘要”走向“可管理系统”

**Mem0** 在这篇综述里非常关键，因为它把文本记忆更新明确化为 ADD、UPDATE、DELETE、NOOP 四类操作。背景是，长期会话里最难的是处理冲突、遗忘和陈旧信息。它解决的痛点是“新信息来了，到底是新增、覆盖、修改还是忽略”，这让记忆不再只是不断 append。
**Memory-R1** 进一步学习这个更新策略，说明研究从规则驱动走向可学习的记忆管理。
**MemTree** 在结构化记忆路线里很重要，它用树来组织信息，并在插入时维护层级摘要一致性。
**MIRIX** 则把人类记忆系统的分层思想引入 agent memory，代表记忆系统从“单一容器”走向“多组件认知式架构”。

这组工作的共同位置是：把 memory 从“长上下文替代品”推进成“可更新、可组织、可推理的个性化基础设施”。 ([arXiv][1])

### 4.5 PRIME / PREFINE / AIPOM / VAIAGE：规划层的典型代表

**PRIME** 的背景是：单靠 profile 或 memory 检索，不足以让 planning 真正“像这个用户一样思考”。它的核心想法是先通过 self-distillation 合成个性化 thought trace，再用这个 trace 约束当前 plan。它的改进点在于，让“过去经验和稳定信念如何影响当前决策”更显式。
**PREFINE** 解决的是 one-shot plan 内部 refinement 问题：即便一轮出计划，也可以先让 pseudo-user critic 按用户 rubric 审一遍，再修正。
**AIPOM** 对应 feedback-driven planning，解决的是用户想直接改中间计划时怎么办。它把计划表示成可编辑结构，支持 mixed-initiative 协作。
**VAIAGE** 则说明个性化 planning 不一定只靠用户反馈，还可以让多个 agent 协商路线可行性与推荐结果。

这几项工作串起来，你能看到 planning 研究从“把用户信息塞进 prompt 里做一轮规划”，发展到“先构造 personalized reasoning prior”，再到“让用户或其他 agent 参与持续修正规划”。 ([arXiv][1])

### 4.6 PEToolLLaMA / PUMA / AWARE-US / ETAPP：执行层的代表

**PEToolLLaMA** 的背景是：很多工具选择问题并不只有“会不会调用”这一层，还涉及“哪个工具更适合这个用户”。它通过交互历史学习 preference-aware tool selection。
**PUMA** 把类似思想放到 web-agent 场景。
**AWARE-US** 非常有代表性，因为它不只做 tool selection，而是处理个性化 grounding 中的经典难点：不可行约束修复。它告诉我们，当约束冲突时，不应随意放宽，而要优先放宽用户最不看重的限制。
**ETAPP** 则进一步把“主动性”纳入 personalization：不是只看是否完成任务，还看 agent 是否会提出真正有帮助、又不过度打扰的额外建议。

这组工作的共同意义是：它们把个性化从“文字层面对齐”推进到了“真实操作和结果层面对齐”。 ([arXiv][1])

### 4.7 图表怎么帮助理解

* **Figure 1**：给出整条闭环。你可以把它理解成“用户请求进入系统后，四个能力模块依次起作用，最后执行结果和反馈再反过来更新用户理解”。它帮助你看到个性化不是单点功能，而是循环系统。
* **Figure 2**：把用户数据分成 historical 与 interaction 两类，帮助理解“双时间尺度个性化”：一部分信息定义用户长期是谁，一部分信息决定此时此刻怎么做。

这些图表的价值不在于视觉上复杂，而在于它们共同把“个性化 agent 是一个闭环系统”这个观点固定下来。 ([arXiv][1])

## 5. 帮你梳理研究发展脉络

这一部分我会更强调“为什么会从上一阶段发展到下一阶段”。

### 5.1 最早的工作在做什么

最早的个性化研究，更多还停留在“让模型知道用户画像或 persona”。也就是从 profile modeling 起步：先把用户特征做成 persona、用户描述或用户特定偏好向量，主要目标是让输出不要那么 generic。这个阶段的问题意识是：**泛化模型缺少个体差异感。** ([arXiv][1])

### 5.2 后来的工作为什么要改进

很快大家发现，仅有 persona 不够，因为它太静态，不能反映用户的动态变化，也不能保留大量交互细节。于是第二阶段转向 memory：不仅要“知道用户是什么样的人”，还要“记住这个用户说过什么、做过什么、最近在意什么、过去改过什么”。问题意识变成：**个性化不应只是静态标签，而应是长期连续关系。** ([arXiv][1])

### 5.3 再后来为什么进入 planning

有了 profile 和 memory 后，研究又意识到一个更深的问题：系统即使知道用户，也不代表能据此做出更好的决策。于是研究重心上升到 planning：用户信息不应只是放在 prompt 里“看着用”，而要显式影响计划生成、子目标分解、约束选择和冲突权衡。问题意识变成：**个性化的关键不是多存一点信息，而是这些信息能不能真正决定行动方案。** ([arXiv][1])

### 5.4 再往后为什么关注 action execution

进一步地，研究者又发现，很多论文做到 plan 这一步就停了，但真实用户感受到的是最终动作和结果，而不是 agent 内部“想得对不对”。于是 action execution 成为下一阶段重点：工具选得对不对、参数填得对不对、失败后会不会修、结果是否按用户偏好排序和呈现。问题意识变成：**个性化必须在外部行为层面可观察。** ([arXiv][1])

### 5.5 每一阶段的关键突破是什么

* **画像阶段的突破**：把用户信息从零散交互变成可复用的内部表示。
* **记忆阶段的突破**：把个性化从单轮现象推进成长期连续过程。
* **规划阶段的突破**：让用户信号真正进入多步决策逻辑。
* **执行阶段的突破**：让个性化从“内部对齐”变成“外部落地”。

这四步正好对应综述的四大能力，也对应我基于综述归纳出的发展路线。 ([arXiv][1])

### 5.6 整个领域的发展趋势是什么

整体趋势是：**从“个性化生成”走向“个性化 agent pipeline”；从静态 personalization 走向动态、持续、闭环 personalization；从只调回答表面，走向影响规划和动作；从单任务对齐，走向跨任务长期协作。** 综述在结论里明确强调，未来应把 personalization 看成 agent 设计的系统属性，而不是单独一个模块。 ([arXiv][1])

### 5.7 现在大家主要关注什么新方向

作者最后总结的新方向很明确：
第一，**decision-critical user modeling**，即不是把所有用户特征都存下来，而是找出对当前决策真正关键的那些；
第二，**temporal dynamics and continual personalization**，即个性化如何长期更新但不灾难性遗忘；
第三，**generalization**，尤其是冷启动用户与跨领域迁移；
第四，**evaluation and benchmarking**，因为当前很多基准仍依赖合成用户和 LLM judge；
第五，**privacy and user control**；
第六，**efficiency and deployment**。
这说明研究已经从“能不能做个性化”进入“怎样把它做得可靠、可评测、可部署”。 ([arXiv][1])

## 6. 比较不同研究路线

下面我做一个横向比较，这部分是我基于综述内容做的整理。

### 6.1 Persona-based vs Response-based 用户建模

**核心思想**：
Persona-based 先概括“这个用户是什么样的人”；Response-based 直接建模“这个用户更喜欢哪类输出”。 ([arXiv][1])

**适用问题**：
Persona-based 更适合长期风格、价值观、行为倾向；Response-based 更适合细粒度输出偏好、候选比较、反馈驱动对齐。 ([arXiv][1])

**优点**：
Persona-based 可解释性高、便于跨任务复用；Response-based 对满意度更直接、细粒度更高。 ([arXiv][1])

**缺点**：
Persona-based 容易过于静态和粗糙；Response-based 依赖高质量反馈，可能不易泛化。 ([arXiv][1])

**复杂度和资源依赖**：
Persona-based 往往实现简单，对显式用户描述和历史日志依赖较大；Response-based 常需要候选输出比较、偏好学习或奖励建模，训练和采样成本更高。 ([arXiv][1])

**适用场景**：
Persona-based 适合长期助理、写作风格对齐、教育陪伴；Response-based 适合推荐、偏好优化、排序、对话 refinement。 ([arXiv][1])

### 6.2 文本记忆 vs 结构化记忆

**核心思想**：
文本记忆把交互压缩成自然语言片段；结构化记忆把信息组织成树、图、triples 等显式关系结构。 ([arXiv][1])

**适用问题**：
文本记忆适合与 LLM 接口自然衔接；结构化记忆适合长距离依赖、复杂关系、跨事件推理。 ([arXiv][1])

**优点**：
文本记忆语义丰富、实现方便；结构化记忆检索更可控、能显式建模关系。 ([arXiv][1])

**缺点**：
文本记忆容易摘要失真和冗余堆积；结构化记忆设计复杂、维护成本高。 ([arXiv][1])

**复杂度和资源依赖**：
文本记忆主要依赖 summarization 和 dense retrieval；结构化记忆还依赖 graph/tree construction、关系更新、结构检索。 ([arXiv][1])

**适用场景**：
文本记忆适合日常会话、轻量助理；结构化记忆适合需要长期关联和复杂知识组织的系统。 ([arXiv][1])

### 6.3 One-shot Planning vs Feedback-driven Planning

**核心思想**：
One-shot 把个性化当先验，一次性出计划；Feedback-driven 把计划当假设，边问边改。 ([arXiv][1])

**适用问题**：
One-shot 适合用户需求相对明确、低延迟要求高的任务；Feedback-driven 适合需求含糊、偏好易变化、代价高的任务。 ([arXiv][1])

**优点**：
One-shot 快、交互负担小；Feedback-driven 对齐更稳、能处理不确定性。 ([arXiv][1])

**缺点**：
One-shot 容易误判隐式偏好；Feedback-driven 增加用户负担和系统延迟。 ([arXiv][1])

**复杂度和资源依赖**：
One-shot 主要依赖好的用户表示和内部反思；Feedback-driven 还依赖交互策略、澄清策略、计划可编辑表示。 ([arXiv][1])

**适用场景**：
One-shot 适合即时规划、低打扰助手；Feedback-driven 适合旅行、协作设计、复杂工作流等高价值决策。 ([arXiv][1])

### 6.4 Action Policy / Grounding / Post-action 三条执行路线

**核心思想**：
Action policy 决定“选什么动作”；action grounding 决定“动作具体怎么落参”；post-action 决定“失败了怎么修、结果怎么呈现”。 ([arXiv][1])

**适用问题**：
Action policy 适合工具选择和执行方式选择；grounding 适合参数补全、约束修复；post-action 适合纠错、重排、主动补充。 ([arXiv][1])

**优点**：
这三层加在一起，使个性化从内部 reasoning 延伸到外部行为。 ([arXiv][1])

**缺点**：
目前研究尚浅，尤其 personalized recovery 还不成熟。 ([arXiv][1])

**复杂度和资源依赖**：
高度依赖具体工具环境，跨环境泛化难。 ([arXiv][1])

**适用场景**：
web agents、委托代理、推荐执行、任务自动化、专业助手。 ([arXiv][1])

## 7. 提炼这个领域已经形成的共识与仍未解决的问题

### 7.1 已经比较明确的结论有哪些

第一，**个性化不是单一模块，而是系统属性**。这几乎是全文最核心的结论。 ([arXiv][1])

第二，**用户数据至少要分历史层和交互层，偏好至少要区分显式/隐式与行为型/主题型**。不同类型的用户信号需要不同处理方式。 ([arXiv][1])

第三，**长期个性化离不开记忆机制**。仅靠一次性 prompt 或静态画像，无法支撑跨轮、跨任务连续性。 ([arXiv][1])

第四，**planning 和 execution 才是个性化真正发挥作用的地方**。如果用户信息只停留在画像层，系统仍然可能做出 generic 决策。 ([arXiv][1])

第五，**评测必须超越 task correctness**，至少要看 effectiveness、adaptivity、generalization、robustness、risk 五个维度。 ([arXiv][1])

### 7.2 还存在哪些争议

一个争议是：**用户模型到底应该多丰富**。模型越丰富，可能越好用，但也更容易引入噪声、隐私问题和无关特征。综述因此提出“decision-critical user modeling”，说明学界也在反思：不是用户信息越多越好，而是要找对当前决策真正关键的那部分。 ([arXiv][1])

另一个争议是：**个性化应当多主动**。系统是否应该频繁提问、主动补充建议、预判用户需求？这在体验和打扰之间存在天然张力。综述在 planning 与 evaluation 中都提到，需要平衡 clarifications 的数量、adaptation efficiency 和 proactivity。 ([arXiv][1])

再一个争议是：**LLM-as-a-judge 到底能否可靠评估个性化。** 因为很多个性化标准本来就主观，而当前基准又高度依赖合成用户与 LLM evaluator。综述认为这会带来 homogenization 和可靠性问题。 ([arXiv][1])

### 7.3 仍未很好解决的问题

目前仍未解决好的问题包括：
持续个性化中的灾难性遗忘；
低交互/新用户上的泛化；
跨领域偏好迁移；
长期 memory 的冗余、陈旧和幻觉；
执行层偏好恢复与执行级评测不足；
隐私、用户控制权和可解释编辑；
以及在真实资源预算下维持有效个性化。 ([arXiv][1])

### 7.4 未来研究可能往哪些方向走

综述总结的未来方向很清楚：
更结构化且决策相关的用户建模；
更强的 continual personalization；
few-shot / meta-learning / uncertainty-aware generalization；
更细粒度、以人为中心的评测；
privacy-aware memory、on-device / federated personalization、用户可编辑记忆；
以及 budget-aware personalization workflow。
这说明未来重点不是“再做一个会记忆的个性化聊天机器人”，而是**把 personalization 做成长期、稳健、可控、可部署的 agent 能力。** ([arXiv][1])

## 8. 用更容易理解的方式再讲一遍

你可以把这篇综述理解成在回答一个很简单但很难的问题：

**为什么现在的大模型助手虽然很强，但总感觉“不够懂我”？如果想让它真的像一个长期合作的人，它需要多出哪些能力？**

这篇综述的答案是：至少要补四层。

### 第一层：先认识你

就像一个新老师第一次带学生，首先要知道这个学生是什么风格：喜欢简洁还是详细，喜欢直接结论还是先铺垫，做事激进还是稳妥。
这就是 **profile modeling**。
有的研究像在给你写“人物小传”，这叫 persona-based；
有的研究不写人物小传，而是看你在两个答案里更喜欢哪个，这叫 response-based。
前者像“认识你的性格”，后者像“认识你的口味”。 ([arXiv][1])

### 第二层：不能认识完就忘

现实里真正有用的助手，不会每次都重新问你“你是不是喜欢正式语气”“你是不是不吃辣”。
所以第二层是 **memory**。
一开始大家只是把历史对话都存起来，但这样太乱。后来有人开始做摘要、分主题、加时间戳；再后来干脆用树、图来组织；再后来不仅会存，还会判断该新增、更新、删除还是忽略。
这说明大家逐渐明白：记忆不是“囤聊天记录”，而是“经营用户关系档案”。 ([arXiv][1])

### 第三层：知道你是谁、记得你说过什么，还不等于会替你做决定

比如你说“帮我做个旅行计划”，真正难的地方不是知道你喜欢海边，而是要把你的预算、时间、节奏、风险偏好都变成一条具体路线。
这就是 **planning**。
有些方法希望尽量不打扰你，一次性给出计划；
有些方法承认“我可能还没搞清你”，于是先问你几个关键问题，再改计划。
前者快，后者稳。
所以 planning 这条线其实是在研究：**什么时候该自己推断，什么时候该向你确认。** ([arXiv][1])

### 第四层：最后得真做出来，而且做法要像你会接受的那样

就算 plan 做对了，执行还可能错。
比如订酒店时，系统可能明明知道你在意预算，但最后选了一个价格更高、位置更好的；或者条件冲突时，它随便放松了一个你其实最在意的约束。
所以最后一层是 **action execution**：
选哪个工具、填哪些参数、失败后怎么修、结果按什么顺序给你看、是否顺手多帮你一步。
这是个性化真正“落地”的地方。 ([arXiv][1])

### 整个领域为什么会这样发展

因为大家逐渐发现：
只调回复语气，不够；
只做用户画像，也不够；
只加个记忆库，还是不够；
真正的个性化，要从“我理解你”一路走到“我替你想”“我替你做”“我做完还按你喜欢的方式交付给你”。
这就是这篇综述最想告诉你的大图景。 ([arXiv][1])

## 9. 最后给你一个高质量总结

### 9.1 这个领域一句话怎么概括

**这篇综述讲的是：如何让 LLM agent 从“泛化地完成任务”进化为“长期、持续、面向具体用户完成任务”，并把个性化贯穿到画像、记忆、规划和执行的整条链路中。** ([arXiv][1])

### 9.2 综述中最核心的 5 类工作是什么

严格按综述本身，最核心是 4 类能力：

1. **Profile Modeling**：建模用户与 agent 角色。
2. **Memory**：长期存储和检索用户相关信息。
3. **Planning**：把用户信号转成个性化决策。
4. **Action Execution**：把决策落到工具与结果上。

如果按你要求扩成 5 类，我基于综述再加上一类：
5. **Evaluation**：因为这篇文章在后半部分明确把评测作为个性化 agent 的独立核心议题来系统整理。 ([arXiv][1])

### 9.3 最值得重点理解的 5 个代表性工作，各自贡献一句话概括

1. **AlignXpert**：把用户偏好表示成高维、可系统分析的 preference space，代表 persona-based 建模从粗标签走向结构化。 ([arXiv][1])
2. **Mem0**：把长期记忆更新显式化为 ADD / UPDATE / DELETE / NOOP，代表 memory 从“存摘要”走向“可管理系统”。 ([arXiv][1])
3. **PRIME**：在 planning 前先生成 personalized thought trace，把记忆和稳定信念真正注入决策过程。 ([arXiv][1])
4. **AIPOM**：把个性化 planning 做成可编辑结构，代表从一次性规划走向 mixed-initiative 协作规划。 ([arXiv][1])
5. **AWARE-US**：把执行时的约束冲突修复变成 preference-aware query repair，代表个性化真正下沉到执行参数层。 ([arXiv][1])

### 9.4 这个领域的发展主线是什么

**从静态用户画像出发，发展到长期个性化记忆；再从“知道用户”发展到“基于用户做规划”；最后从内部推理扩展到外部执行与结果交付；同时评测也从“答对没”发展到“是否对这个用户长期有效、可适应、能泛化、够稳健、且风险可控”。** ([arXiv][1])

### 9.5 如果你要继续深入读原始论文，优先从哪些工作开始

如果你的目标是建立这条脉络，我建议按这个顺序读：
**AlignXpert / FSPO → Mem0 / MemTree → PRIME / PREFINE → AIPOM / VAIAGE → PEToolLLaMA / AWARE-US / ETAPP。**
这样读的好处是，你会顺着这篇综述的主线，从“用户怎么表示”一路读到“个性化怎么真正执行出去”，而不是只在某个局部打转。 ([arXiv][1])
